# Module 57 — Account Profile, Notification Center & Settings Completion

## Purpose

Module 57 makes account/profile, notifications, and shop settings feel production-ready with real persistence and scoped data. It avoids fake notification counts, fake profile fields, and unsupported settings controls.

## Account/profile features

- Added authenticated account endpoints for profile, password, and preferences.
- Profile exposes safe user data only: name, email, phone, role, status, timestamps, and preferences.
- Email is read-only in the frontend because a verified email-change flow is not implemented.
- Users can update their own name and phone.
- Password updates require the current password and a confirmed new password.
- Preferences save to `users.preferences_json` with language, timezone, date format, dashboard range, and notification preference flags.

## Notification center design

- The notification center uses real `notification_logs` generated by existing Telegram/order/payment flows.
- The navbar bell shows a real unread count when authenticated.
- The dropdown shows the latest real notification logs and links to the full page.
- The full page supports All, Unread, Orders, Payments, and System filters.
- Mark-as-read and mark-all-as-read use per-user read records in `notification_log_reads`.

## Notification backend strategy

- Reused existing `notification_logs` instead of creating fake notification data.
- Added per-user read state through `notification_log_reads` so one user reading a notification does not mark it read for another user.
- Notification payloads intentionally omit recipient, provider secrets, raw payment payloads, and unapproved metadata.
- Supported notification categories are inferred from real event names:
  - `order.*` → Orders
  - `payment.*` and `invoice.*` → Payments
  - everything else → System

## Account menu integration

- Account menu links now include Profile, Shop Settings, System Health, Notifications, and Logout.
- Settings and System Health remain permission-aware.
- Profile and Notifications are real routes and not placeholders.

## Settings completion improvements

- Settings page keeps the existing supported save fields for shop profile, branding, operations/billing, and Telegram notifications.
- Added Payment Settings as an honest read-only readiness section because shop-specific payment toggles are not supported yet.
- Added Public QR Menu section with saved display name, public status, language behavior, and preview link.
- Added setup completion card in the settings sidebar.

## Completion score logic

The score uses only data retrieved from existing APIs:

- Shop profile complete: name, phone, address, and description exist.
- Branch exists: `/shops/{shop}/branches` returns at least one branch.
- Category exists: `/shops/{shop}/categories` returns at least one category.
- Product exists: `/shops/{shop}/products` returns at least one product.
- Table QR exists: `/branches/{branch}/tables` returns at least one table.
- Admin notification channel ready: Telegram enabled and chat ID saved.
- Public QR menu ready: shop is active and has category/product content.

## Security rules

- Account endpoints require Sanctum authentication.
- Users can update only their own profile/preferences/password.
- Profile update ignores role/status changes.
- Password hashes and remember tokens are never returned.
- Notification queries are scoped to shops/branches accessible to the authenticated user.
- Notification read state is user-specific.
- Settings update authorization remains owner/super-admin only.
